# Dev Log

## 2026-05-05

### Project Direction
- Building a web app for research-assisted writing across blogs, articles, letters, reports, and docs.
- Core workflow:
  1. Create a project.
  2. Upload markdown resources organized in folders/subfolders.
  3. Read rendered markdown resources without editing them.
  4. Create quote-anchored notes/highlights.
  5. Review visible AI expert notes/suggestions tied to source context.
  6. Accept, discard, or edit AI suggestions into user-owned notes.
  7. Generate a report draft from notes.
  8. Edit the report as paragraph blocks.
  9. Use per-block AI actions for tone alternatives and critique/challenge.
  10. Export the finished report as markdown.

### Confirmed Product Decisions
- `v1` is single-user.
- First product slice is the reading workspace, not the full AI/editor stack.
- Resource storage is backend-managed through a storage abstraction.
- Use local disk for `v1` persistence and temporary files.
- Keep the storage layer abstract so `v2` can move to object storage such as MinIO.
- If needed later, add a cache layer such as Redis when scale/performance justifies it.
- Resource ingestion in `v1` is web UI upload.
- Resources are immutable source material after upload.
- The reading surface in `v1` is rendered markdown.
- Annotation scope in `v1` is quote-only notes/highlights.
- AI-generated notes must be visible and reviewable in the UI, not hidden system output.

### Architecture Direction
- Recommended architecture: modular monolith with async AI/background jobs.
- Planned backend modules:
  - `projects`
  - `resources`
  - `annotations`
  - `ai_suggestions`
  - `reports`
  - `exports`

### Approved V1 Report Update Rule
- Allow new markdown uploads later within an existing project.
- Allow reading and note-taking on those newly added resources.
- Do not let `v1` surgically rewrite an existing report block-by-block based on new notes.
- Instead, if a report already exists and new notes arrive, mark the report as stale/outdated and let the user explicitly generate a new draft version from the full current note set.
- Reserve surgical AI updates to targeted report blocks for a later version once note-to-block mapping and review UX are well-defined.

### Approved Architecture Notes
- Backend/storage section approved as `lgtm`.
- `v1` uses local disk for persistence and temp files.
- Storage abstraction remains in place so future versions can move to MinIO/object storage and add caching if needed.

### Approved Reading Workspace Behavior
- The core reading workspace has three primary surfaces:
  - `project/resource tree`
  - `rendered document viewer`
  - `notes review panel`
- Resources remain read-only after upload.
- Highlights and notes are created from rendered markdown, not raw markdown source.
- Annotations are quote-anchored and store enough metadata to relocate the source span later.
- User notes and AI suggestions are separate by default.
- AI suggestions are visible review items grouped by lens/expert and tied to source context.
- Accepting an AI suggestion converts it into a user-owned note while preserving provenance.
- Discarding an AI suggestion removes it from the active workflow but keeps a reversible record.
- Report generation uses only user-owned notes, including AI suggestions the user explicitly accepted.

### Approved Report And Editor Behavior
- Report generation uses the current set of user-owned notes as its source material.
- The initial report draft is generated by the AI system, not manually assembled by the user.
- Each draft is stored as a distinct `report version`.
- After draft generation, the product shifts into an editor mode.
- The editor is organized as paragraph blocks.
- Manual typing is the default editing path, and user edits always take precedence.
- Per paragraph block, the user can request:
  - `change tone` suggestions
  - `challenge my argument` critique
- Editor-phase AI output is shown as suggestions, not silently applied changes.
- Export downloads the current report version as markdown.
- If new resources or notes are added later, the current report version is marked stale and the user explicitly generates a revised draft version.

### Approved Data Flow And Failure Handling
- Uploading markdown creates immutable `resource` records and stores files on disk through the storage abstraction.
- Opening a resource renders it for reading/highlighting without mutating the original markdown.
- Creating a highlight/note stores a quote anchor and metadata linking the note back to a specific resource and span.
- Running AI analysis creates background suggestion jobs that persist AI suggestions when they complete.
- Accepting a suggestion creates or updates a user-owned note.
- Discarding a suggestion changes suggestion state only and does not touch the source resource.
- Generating a report creates a report-generation job that reads only user-owned notes and writes a new report version.
- Per-block editor AI actions return suggestions without mutating the block unless the user explicitly applies one.
- If upload or parsing fails for one file, that file is marked failed while the rest of the batch can still succeed.
- If an AI job fails, the failure is visible in the UI and offers a retry action.
- If report generation fails, no partial draft replaces the current report, and the user can retry generation with an explicit retry button.
- If new notes arrive after draft creation, the report is marked `stale`, but nothing is auto-regenerated.
- Destructive-looking actions on AI suggestions should be reversible where practical.
